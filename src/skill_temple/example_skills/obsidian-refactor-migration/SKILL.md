# Obsidian Refactor Migration

## Purpose
把 Obsidian 中高风险的结构变更——重命名、移动、拆分、合并、归档、属性/标签迁移——变成可分批、可验证、可回退的流程。

本 skill 的核心不是“多做变更”，而是“缩小变更批次、保持链接完整、优先归档而不是删除”。

## When to use
在以下情况使用本 skill：

- 用户要批量重命名、移动、归档或整理笔记。
- 用户要把扁平目录迁移为新的结构。
- 用户要把旧属性名迁移为新属性名，或把标签体系迁移为属性体系。
- 用户要拆分超大笔记、合并重复笔记、清理旧项目、关闭已完成事项。
- 用户要做“长期维护型整顿”，而不是单次小编辑。

## When not to use
以下场景不要使用本 skill：

- 单次新增或局部更新内容；那是 `obsidian-note-router`。
- 全局规范尚未定义时就想直接大迁移；应先使用 `obsidian-vault-governance`。
- 只是想评估问题，不立刻执行调整；先用 `obsidian-consistency-audit`。
- 只是需要 Markdown / Base / Canvas 语法帮助；应交给现有语法 skill。

## Scope
### 负责
- 制定并执行有边界的重构或迁移批次。
- 处理笔记拆分、合并、重命名、移动、归档、属性迁移、标签迁移、附件配套迁移。
- 检查受影响的链接、别名、搜索、视图、查询、任务与索引页面。
- 输出迁移映射表、验证步骤、完成日志。

### 不负责
- 不制定一套全新治理规范。
- 不做持续性的体检扫描。
- 不把历史内容一次性全部“洗白”。
- 不在缺少明确目标状态时盲目批量改动。

## Workflow
1. **确认目标状态**：来自既有治理规范，或先抽出最小目标规则。
2. **划定作用域**：只选一个目录、一类笔记、一组属性、一个项目批次，不做全 vault 混合迁移。
3. **建立影响面清单**：
   - 文件本身
   - 内链 / 反链 / alias
   - 属性 / 标签 / 模板
   - Bases / Dataview / Canvas / 仪表板
   - 任务与附件
4. **选择变更策略**：重命名、移动、拆分、合并、归档、映射迁移。
5. **先做可逆动作**：能归档的不删除，能 alias 的不硬断旧名，能分批的不一次改完。
6. **逐批验证**：每一批结束后检查链接、搜索结果、视图、任务和关键索引。
7. **记录迁移日志**：保留变更映射与未完成项。

## Decision rules
### 1. 先小批次，再扩展
任何重构都要先从小范围试点开始：
- 一个目录
- 一类笔记
- 一个属性
- 一个项目集合

不要在同一批中同时做：
- 改目录
- 改命名
- 改属性 schema
- 改模板
- 改视图层

至少先把一类变更稳定，再做下一类。

### 2. 重命名优先于重建
官方文档说明 Obsidian 可自动更新内部链接，CLI 的 `move` / `rename` 也会在该设置启用时自动更新链接。

因此：
- 想改变 canonical 名称时，优先重命名原笔记。
- 只有在内容职责真的改变时，才新建并迁移内容。
- 旧名称仍有检索价值时，加进 `aliases`，而不是保留一份平行旧笔记。

### 3. 拆分笔记的条件
满足以下任一条件时，优先拆分：
- 一个笔记里混入多个独立实体。
- 一个笔记同时承载项目总览、会议全文、资料摘录、任务仓库等多个生命周期。
- 某些 section 已经值得被单独链接、单独归档、单独回顾。

拆分后：
- 原笔记保留索引角色或摘要角色。
- 新子笔记与原笔记互相链接。
- 不要把原笔记删空到失去导航价值，除非用户明确要求。

### 4. 合并笔记的条件
只有在确认“它们是同一 canonical entity”时才合并。

合并前先检查：
- 是否只是同名异物。
- 是否一个是资料笔记，一个是结论笔记。
- 是否两个笔记的生命周期其实不同。

如果只是同一对象的不同叫法：
- 首选保留一个主笔记。
- 旧名转 alias 或在迁移期保留简短 redirect note。

### 5. 归档优先于删除
官方文档说明删除行为可以进系统回收站、Obsidian `.trash` 或永久删除。

因此：
- 对已完成项目、过期资料、旧实验、旧阶段日志，默认归档。
- 只有明确低价值、可再生、无引用风险的 temp/test 产物，才考虑删除。
- 默认不要永久删除可能仍有参考意义的知识与项目记录。

### 6. 属性或标签迁移要有映射期
官方文档说明 properties 可被全局管理和重命名，Bases / Search / Templates 也依赖属性。

因此迁移字段时：
- 先定义 `old -> new` 映射表。
- 允许短期共存，不要一步切断所有旧引用。
- 清点所有 Base / Dataview / 模板 / 搜索语句的依赖后再收尾。

### 7. Base / Dataview / Canvas 是受影响面，不是迁移动力源
- Bases 依赖 properties 和文件。
- Dataview 依赖查询来源和字段。
- Canvas 依赖文件路径与节点引用。

因此：
- 迁移时要同步审查这些视图层是否失效。
- 但不要为了保留某个旧视图的脆弱写法而阻止更合理的主数据层迁移；应更新视图。

## Best practices
### 推荐做法
- 所有批量变更都保留映射表：旧路径 → 新路径，旧属性 → 新属性。
- 归档闭环项目时，保留简短结束说明或索引页，而不是彻底消失。
- 拆分大笔记时保留顶部摘要与去向链接。
- 重命名后将旧名纳入 alias，保持检索连续性。
- 任何 schema 迁移都先检查模板、仪表板、搜索语句和任务面板。

### 可选做法
- 对高风险变更先做 dry-run 计划，只输出批次与影响面，不立即执行。
- 对旧项目使用“软关闭”：转入 archive，保留少量活动字段供回溯。
- 对大型 vault 把迁移按目录分季度进行，而不是一次完成。

### 不建议做法
- 全 vault 盲目全局替换术语。
- 在同一波迁移里同时做语义重命名和结构重排。
- 直接永久删除历史项目、历史资料和旧决策。
- 把合并当作清理重复的默认策略；很多“重复”其实是不同生命周期的并存笔记。

更多清单见 [references/migration-checklists.md](references/migration-checklists.md)。

## Safety rules
- 改名前先确认是否存在同名异物或 alias 冲突。
- 移动附件前先检查 embed 是否仍能解析。
- 删除前先确认不是 Base / Canvas / Dataview / 任务面板的隐式依赖。
- 任何批次出现高不确定性时，立即缩小范围，而不是继续推进。
- 对用户高度个性化的历史结构，优先保留可读性与可追溯性，不追求绝对统一。

## Output contract
本 skill 的产出至少应包含：

1. **迁移目标与作用域**
2. **批次划分**
3. **变更映射表**：旧名/旧路径/旧字段 → 新名/新路径/新字段
4. **受影响面清单**：链接、模板、Bases、Dataview、Canvas、任务、附件
5. **执行顺序**
6. **每批验证项**
7. **未完成项与风险项**
8. **迁移日志或完成记录**

## Failure modes
### 失败模式 1：为了整齐而过度搬迁
处理：优先解决检索、维护、重复、链接失效等真问题；不要为审美做大迁移。

### 失败模式 2：拆分后失去导航
处理：原笔记保留索引职责，给新笔记加回链。

### 失败模式 3：字段迁移后视图层失效
处理：把 Base / Dataview / 模板 / 搜索语句纳入必检清单。

### 失败模式 4：把 archive 当 trash
处理：archive 仍是可检索、可回顾、可引用的长期区；真正无价值的短命文件才进删除流程。

## Examples
### 示例 1：归档已完成项目
用户想把去年完成的项目全部移出 active 区。
本 skill 应分批归档，保留项目索引或结项摘要，不直接删除。

### 示例 2：属性迁移
用户想把 `stage` 统一改成 `status`。
本 skill 应先清点所有搜索、模板、Bases、Dataview 依赖，再按映射分批迁移。

### 示例 3：拆分大项目笔记
用户的项目主页已经塞满会议、任务、资料摘录。
本 skill 应拆出 meeting / reference / review 子笔记，主笔记改为索引与决策总览。

### 示例 4：改名并保留旧称呼
用户想把 `AI Notes` 统一改成 `LLM Notes`。
本 skill 应优先重命名主笔记并添加 alias，而不是新建平行笔记。
