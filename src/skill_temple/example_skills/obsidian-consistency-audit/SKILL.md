# Obsidian Consistency Audit

## Purpose
对 Obsidian vault 做“先发现问题、再决定要不要动手”的一致性审计。

本 skill 的目标不是立刻改，而是识别哪些不一致会真正损害检索、链接、视图、任务管理和 AI 协同，哪些只是无害差异，不值得大动干戈。

## When to use
在以下场景使用本 skill：

- 用户想知道现有 vault 是否适合继续让 AI 协同写入。
- 用户怀疑存在重复笔记、属性污染、标签膨胀、命名漂移、临时垃圾或归档失控。
- 用户准备做一次迁移或整理，但想先做基线体检。
- 用户想评估某套模板、任务规则或结构化内容规则是否已失效。
- 用户想周期性维护 vault，而不是凭感觉清理。

## When not to use
以下场景不要使用本 skill：

- 用户只想创建或更新某个具体笔记。
- 用户已经明确要执行迁移方案；那时交给 `obsidian-refactor-migration`。
- 用户正在定义新治理规范；那时优先 `obsidian-vault-governance`。
- 只是路由一条新输入；那时用 `obsidian-note-router`。

## Scope
### 负责
- 发现命名、目录、标签、属性、模板、任务、链接、归档、临时文件方面的漂移。
- 识别 AI 生成内容可能带来的污染与维护负担。
- 输出证据、严重度、影响面、修复建议序列。
- 明确哪些问题不需要处理，避免过度整理。

### 不负责
- 不直接批量修改内容。
- 不假定整个 vault 必须高度统一。
- 不把个性化写作风格视为错误。
- 不把所有历史遗留都当成必须清理的技术债。

## Workflow
1. **收集基线**：已有治理规范、显式命名规则、主要笔记类型、常用模板、社区插件依赖。
2. **选择审计范围**：全 vault 或某一层级（例如 `Projects`、`References`、`Daily`）。
3. **按维度审计**：
   - 命名规范
   - 文件夹落点
   - 标签体系
   - 属性名称 / 类型 / 空值情况
   - 链接与反链
   - 模板使用效果
   - 任务放置与状态一致性
   - temp / test / scratch 残留
   - archive 边界
   - Bases / Dataview / Canvas 对主结构的依赖
4. **分级**：
   - `high`：会直接伤害搜索、链接、视图、任务或 AI 判断。
   - `medium`：会持续制造混乱，但短期不致命。
   - `low`：主要是观感或局部不一致。
5. **输出修复建议序列**：先修高收益、低风险项。
6. **明确不处理项**：说明为什么无需处理，防止“为了统一而统一”。

## Decision rules
### 1. 先抓会伤害检索和路由的漂移
优先关注：
- 同义属性并存，如 `status` / `state` / `stage`
- 标签大小写或命名变体造成的散裂
- 重复笔记或同名异物
- 项目 / 资料 / 日记 / 临时文件混放
- orphan durable notes（看似正式但无有效连接）

官方文档说明 properties 可被 Search、Templates、Bases 使用，而 tags、links、backlinks 也直接影响检索和网络结构。

因此：
- 只要某种不一致会影响搜索表达、Base 字段、模板合并、链接网络，就优先级更高。

### 2. 把“重复风险”放在高优先级
以下情况优先排查：
- 近义标题、别名未利用、平行新旧名称。
- 同一来源被多个 reference note 重复吸收。
- 同一项目存在多个“主页”。
- AI 生成的摘要笔记与原始资料笔记长期并存但无明确分工。

### 3. 不把所有差异都判为问题
以下情况可能是合理差异：
- 不同笔记类型本来就有不同属性集。
- archive 区保留旧命名风格是可接受的。
- daily notes 与 permanent notes 的结构不同是正常现象。
- 多语言 vault 的命名差异未必是错误。

### 4. 临时文件残留要结合生命周期判断
官方文档说明可配置默认新笔记位置、附件位置与 excluded files；这为隔离临时区提供了原生手段。

因此：
- 重点不是“有无 temp 文件”，而是它们是否越界进入正式区、是否长期不清理、是否影响搜索与路由。

### 5. 把 AI 污染识别为独立审计维度
出现以下模式时，判定为 AI 协同风险：
- 无来源、无时间标记的 AI 摘要直接混入用户原文。
- 大量内容空泛但正式落盘。
- 新增属性名和标签名呈爆炸式增长。
- 同一主题反复新建相似笔记，没有复用与链接策略。

### 6. 视图层问题要追到主结构
- Base 字段缺失，往往是 properties 规范失效。
- Dataview 报表漂移，往往是来源目录或字段命名漂移。
- Canvas 关系图失真，往往是主文件未维护或文本卡片长期未转文件。

因此审计时不要只修视图；应找出背后的主结构问题。

## Best practices
### 推荐做法
- 每次审计都输出“问题 + 证据 + 严重度 + 建议动作 + 是否值得做”。
- 把受控词表、必需属性、笔记类型矩阵当作审计标准，而不是凭印象判断。
- 审计前先把 active 区、archive 区、temp 区分开看，避免历史区噪声淹没当前问题。
- 优先提出小批次、低风险的修复建议。

### 可选做法
- 对大型 vault 只抽样关键区域，不强求全量扫描。
- 先做“只读审计”，待用户确认后再转入迁移 skill。
- 对项目型 vault 可增加“闭环性”审计：项目是否有 owner、status、next action、review date。

### 不建议做法
- 以“统一”为名清除所有个体差异。
- 因为看起来不整齐就重构整个 archive。
- 没有证据就把某类笔记标为重复或垃圾。
- 审计报告只给结论不给证据，导致无法执行。

详细 rubric 见 [references/audit-rubric.md](references/audit-rubric.md)。

## Safety rules
- 在没有充分证据前，不建议合并、删除或大迁移。
- 对每条发现都要给出置信度与样本证据。
- 明确区分“建议处理”和“无需处理”。
- 对用户已有高自由度写作区，不以企业数据库标准强压一致性。

## Output contract
审计结果至少应包含：

1. **审计范围与假设**
2. **总体结论**：是否适合继续 AI 协同写入
3. **按维度的主要发现**
4. **每条发现的证据、严重度、影响**
5. **建议修复顺序**
6. **明确不建议动的部分**
7. **若需后续行动，应交给哪个 skill**

## Failure modes
### 失败模式 1：把多样性误报成混乱
处理：先按笔记类型和目录分层，再判断是否漂移。

### 失败模式 2：只抓小毛病，忽略主问题
处理：优先报告影响搜索、链接、视图、路由和维护成本的问题。

### 失败模式 3：报告不可执行
处理：每条问题必须配修复建议和推荐 skill，而不是只给抽象评价。

### 失败模式 4：归档区噪声干扰结论
处理：把 archive 作为单独审计层，不让其覆盖 active 区判断。

## Examples
### 示例 1：检查属性污染
用户发现同类项目笔记的字段越来越乱。
本 skill 应识别近义属性、类型不一致、空值泛滥，并给出修复优先级。

### 示例 2：检查重复笔记
用户怀疑 AI 过去半年制造了大量重复资料笔记。
本 skill 应基于标题、alias、来源、链接关系和内容职责给出重复嫌疑清单，而不是直接合并。

### 示例 3：检查 temp 残留
用户想知道测试文件是否污染了正式 vault。
本 skill 应检查 temp/test/scratch 是否越界到主目录，是否仍被搜索与链接网络感知。

### 示例 4：检查治理是否失效
用户已经有模板和命名规范，但感觉越来越不遵守。
本 skill 应对比规范与现状，指出漂移最明显的类型和最先修的点。
