# Obsidian Note Router

## Purpose
把“收到一段新信息后，AI 到底应该写到哪里、怎么写、写多少”变成可重复执行的决策流程。

本 skill 的目标不是格式生成，而是路由与最小安全写入：先搜索，再判断是更新旧笔记、新建笔记、创建临时笔记，还是只输出提案不落盘。

## When to use
在以下场景使用本 skill：

- 用户给出一段信息，希望加入 vault，但没有明确指出目标笔记。
- 用户说“帮我记一下/整理一下/放进 Obsidian”，但没有说明是新建还是追加。
- 用户要把网页、会议纪要、阅读摘录、灵感、项目进展等写入 vault。
- 用户让 AI 代写内容，但你需要先判断是否会造成重复笔记或污染现有笔记。
- 你需要决定“搜索优先 vs 新建优先”“追加到旧笔记 vs 分裂成新笔记”。

## When not to use
以下场景不要使用本 skill：

- 用户已经明确给出目标文件路径与改动范围，且重复风险很低。此时可直接用 `obsidian-markdown` 或 `obsidian-cli` 执行。
- 用户要设计全局结构、属性字典、命名规范；那是 `obsidian-vault-governance`。
- 用户要批量整理、迁移、归档、改名；那是 `obsidian-refactor-migration`。
- 用户要做周期性规范审计；那是 `obsidian-consistency-audit`。

## Scope
### 负责
- 搜索候选笔记并判断是否已存在对应实体。
- 判断是更新现有笔记、创建新笔记、创建 inbox/scratch/temp 笔记，还是只生成提案。
- 为新笔记选择合适类型、落点、命名、最小属性、最小链接。
- 控制写入范围，避免整篇重写和无意义规范化。

### 不负责
- 不制定全局治理规范。
- 不执行跨 vault 的批量重构。
- 不做全量体检报告。
- 不重复教授 Markdown、Bases、Canvas 语法。

## Workflow
1. **识别输入类型**：新增事实、项目更新、会议记录、资料摘录、临时验证结果、任务、想法。
2. **先搜索候选**：
   - 按标题、别名、标签、属性、项目名、日期、来源名、主题词搜索。
   - 必要时查看 backlinks / unlinked mentions / related notes。
3. **给候选分级**：
   - `exact match`：已存在同一实体的主笔记。
   - `related note`：主题相关但不是同一实体。
   - `no good match`：适合新建。
4. **做路由决策**：
   - 更新现有笔记
   - 新建 durable note
   - 新建 inbox note
   - 新建 temp note
   - 不落盘，仅输出建议
5. **确定最小写入策略**：
   - 追加新 section
   - 局部改写特定 section
   - 新建子笔记并回链
   - 仅记录任务或来源，不动主内容
6. **补足最小上下文**：命名、属性、至少一条有意义链接、必要的来源标识。
7. **写入后复核**：确认没有制造重复笔记、平行命名、错误属性或临时垃圾。

## Decision rules
### 1. 默认先搜索，再决定写入
官方文档说明 Search 支持全文搜索、属性搜索、标签搜索、任务搜索；Backlinks 和 Outgoing links 还能帮助发现 linked mentions 与 unlinked mentions。

因此：
- 只要用户输入涉及已有项目、已有人名、已有主题、已有来源、已有日期对象，就先搜索候选。
- 未搜索就新建，属于高风险操作。

### 2. 满足“同一实体、同一生命周期、同一主记录”时才追加
把新内容追加到旧笔记，必须同时满足：
- 它描述的是同一对象，而不是相关对象。
- 原笔记本来就是该对象的主记录。
- 追加不会把笔记混成多种类型。

例如：
- **应追加**：给已有项目主页更新本周状态。
- **不应追加**：把某次会议完整纪要塞进项目总览；更适合新建会议笔记并回链到项目。

### 3. 满足“新实体、新事件、新来源、新时间片”时优先新建
以下情况默认新建：
- 新的会议 / 新的文献 / 新的网页来源 / 新的客户 / 新的项目 / 新的一天的日记。
- 旧笔记已是稳定 evergreen note，而新内容只是当天过程或一次性材料。
- 同一主题下出现独立生命周期的材料。

### 4. 不确定时优先缩小影响面
如果无法稳妥判断“该不该改旧笔记”：
- 优先新建 linked note 或 inbox note。
- 或只生成建议，不直接改动主笔记。
- 避免把不确定信息直接混入长期主记录。

### 5. 用笔记类型决定落点
默认路由矩阵见 [references/routing-matrix.md](references/routing-matrix.md)。

核心原则：
- **daily**：当天日志、碎片、工作轨迹、当天捕获。
- **project**：持续协调、里程碑、决策、总览。
- **meeting**：单次会议，按日期或会议主题独立建笔记。
- **reference**：来源笔记、阅读摘录、网页清洗内容。
- **permanent**：稳定结论、抽象知识。
- **inbox**：暂时无法判断类别，但值得保留。
- **temp**：仅为验证、试写、转换、OCR、实验而生的短命文件。

### 6. 临时文件必须有短生命周期
官方文档说明可通过 Files and links 设置默认新笔记位置、默认附件位置，也可通过 Excluded files 隐藏不需要参与搜索和链接建议的区域。

因此：
- 临时测试笔记、转换中间文件、AI 试写稿、验证脚本产物要放入专门 temp 区。
- temp 区应有明确清理规则；不应与正式知识区混放。
- 默认不要让临时产物进入主导航、主搜索和主链接网络。

### 7. 追加时优先新增区块，而不是重写旧内容
- 对用户已有笔记，优先追加明确标题区块，如 `## AI 摘要`、`## 本周更新`、`## 新增来源`。
- 除非用户要求重写全文，否则不要改写用户原文的主体叙述。
- 若新增内容来自外部资料，应写明来源或日期，避免把 AI 推断伪装成用户原始笔记。

### 8. 需要 durable note 时，至少留下一个有效连接点
官方文档说明链接、别名、反向链接和未链接提及共同构成知识网络。

因此新建长期笔记时：
- 至少连向一个父项目、父主题、来源、MOC 或相关笔记。
- 若存在 canonical index，也要从 index 回链或纳入列表。
- 不要创建完全孤立但名字看起来“像正式笔记”的内容。

## Best practices
### 推荐做法
- 先判断笔记类型，再决定路径、命名、模板。
- 用 alias 吸收旧称呼，而不是重复建笔记。
- 用最小属性集，避免每次新建都发明新字段。
- 来源型内容先沉淀为 reference note，再决定是否抽取到 permanent note。
- 项目动态优先写进项目 note 或其子笔记，不直接散落为多个无主笔记。

### 可选做法
- 对未确定去向的高价值输入，先进入 inbox，再由人工或后续技能二次分流。
- 对网页内容，可先用现有 `defuddle` 清洗正文，再路由到 reference note。
- 对同一天大量碎片输入，可先写 daily，再在周回顾或项目回顾时提炼。

### 不建议做法
- 不搜索就新建正式笔记。
- 为一次性输入创建看似永久的正式知识笔记。
- 在项目主页里塞入所有会议全文、所有资料全文、所有临时记录。
- 每次轻触旧笔记都顺手改命名、补属性、改标签，导致写入范围失控。

## Safety rules
- 如果重名或近义名很多，默认不做破坏性覆盖。
- 如果找不到足够强的匹配证据，不要把新内容硬追加进旧笔记。
- 如果内容只是试验、推理草稿、转换中间产物，默认进入 temp 而不是正式区。
- 如果要改用户既有正文，尽量限定到一个 section 或一个 patch。
- 如果用户结构不明，优先沿用现有模式，而不是引入新目录、新字段、新模板。

## Output contract
本 skill 的结果至少要包含：

1. **路由结论**：更新旧笔记 / 新建 durable note / 新建 inbox / 新建 temp / 不落盘
2. **目标路径或目标类型**
3. **为何这样路由**
4. **拟写入范围**：追加 section、局部改写、整篇新建、仅创建任务等
5. **最小补充结构**：标题、属性、链接、来源字段、模板名（如适用）
6. **风险提示**：是否存在重名、重复、混型、临时污染风险

## Failure modes
### 失败模式 1：把“相关笔记”当成“同一笔记”
处理：提高匹配门槛；不确定时新建 linked note。

### 失败模式 2：正式内容与临时内容混写
处理：分出 temp/inbox 层，避免试验稿进入正式知识区。

### 失败模式 3：项目主页过载
处理：会议、资料、周报、决策分拆为子笔记，再由主页聚合。

### 失败模式 4：AI 生成内容伪装成人类原文
处理：为 AI 生成段落加清晰区块标题、时间或来源标记。

## Examples
### 示例 1：导入网页摘要
用户发来一篇文章并说“放进我的 Obsidian”。
本 skill 应先搜索是否已有该来源笔记；若无，创建 reference note，而不是直接塞进某个 permanent note。

### 示例 2：更新项目状态
用户说“把今天的进展记到 Alpha 项目里”。
本 skill 应先搜索 `Alpha` 主项目笔记；若存在且该笔记承担项目总览职责，则追加 `## 本周更新` 或 `## 今日进展` 区块。

### 示例 3：记录会议
用户给出 2026-05-13 的会议纪要。
本 skill 应新建 meeting note，并从项目主页或日记中回链，而不是把会议全文追加到项目主页正文。

### 示例 4：处理不确定灵感
用户给出一段模糊的想法。
本 skill 应优先落到 inbox 或 daily，而不是仓促创建 permanent note。
